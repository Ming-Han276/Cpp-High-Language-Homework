## 四、主要流程

### 1\. 整体流程

游戏整体采用状态机设计，分为三个核心运行状态，流程如下：

1. **开始界面状态**

    - 程序启动后自动加载所有游戏资源（背景图、玩家素材、障碍物素材）

    - 显示游戏标题动画、难度选择按钮、音乐控制按钮、退出按钮

    - 玩家可选择游戏难度，点击开始按钮进入游戏环节

2. **游戏运行状态**

    - 进入 60FPS 固定帧率的游戏主循环

    - 每帧依次执行：输入处理→玩家状态更新→障碍物生成与更新→碰撞检测→画面绘制

    - 若检测到碰撞则切换至结束界面，否则循环运行

3. **结束界面状态**

    - 计算并展示玩家的最终存活得分

    - 提供重新开始游戏、返回主页的功能按钮

    - 保留音乐控制与退出功能，支持玩家再次游玩

### 2\. 算法或公式

项目中实现了多个核心算法支撑游戏运行，具体如下：

#### （1）跳跃物理模拟

采用简化的重力加速度模型模拟玩家跳跃的物理过程，公式如下：

```cpp
pY += jumpVel;          // 更新玩家垂直位置
jumpVel += gravity;     // 速度叠加重力加速度，模拟重力效果
```

其中重力常量`gravity=2`，初始跳跃速度`jumpVel=\-22`（向上为负方向），当玩家落地后自动重置跳跃状态，保证跳跃过程流畅自然。

#### （2）AABB 碰撞检测

采用轴对齐包围盒（AABB）算法实现玩家与障碍物的碰撞检测，通过判断两个矩形的边界重叠情况判定碰撞：

- X 轴重叠判定：`\!\(playerRight \&lt; obsLeft \|\| playerLeft \&gt; obsRight\)`

- Y 轴重叠判定：`\!\(playerBottom \&lt; obstacleTop \|\| playerTop \&gt; obstacleBottom\)`
仅当两个轴同时满足重叠条件时，才判定为碰撞。同时针对不同玩法做了定制优化：

- 地面障碍物：玩家处于跳跃状态时，自动忽略碰撞，实现跳跃躲避

- 头顶障碍物：玩家处于下蹲状态时，自动忽略碰撞，实现下蹲躲避

#### （3）障碍物生成算法

为保证游戏的合理性与可玩性，障碍物生成采用了以下规则：

1. 随机打乱三个车道的顺序，优先在有空位的车道生成障碍物

2. 同车道的障碍物之间保证最小间隔`OBSTACLE\_GAP=240`，避免障碍物重叠

3. 头顶障碍物会提前 1\.5 秒显示警告线，提醒玩家提前躲避

4. 不同难度对应不同的障碍物移动速度：简单难度 9、中等难度 13、困难难度 17，难度越高游戏挑战越大

#### （4）帧率控制

为保证游戏在不同配置设备上运行速度一致，实现了固定 60FPS 的帧率控制：

```cpp
const int FPS = 60;
const DWORD frameDelay = 1000 / FPS;
DWORD cost = GetTickCount() - frameStart;
if (cost < frameDelay) Sleep(frameDelay - cost);
```

每帧计算运行耗时，若未达到目标帧时间则休眠剩余时间，保证游戏运行流畅稳定。

### 3\. 单元测试设计

针对各个功能模块，设计了对应的单元测试案例，验证功能的正确性，具体测试设计如下：

- 输入处理测试：验证键盘换道、跳跃、下蹲操作，以及鼠标按钮点击的响应是否正确

- 物理更新测试：验证跳跃的重力模拟、落地重置的逻辑是否正确

- 碰撞检测测试：验证不同状态下的碰撞判定、躲避逻辑是否符合预期

- 障碍物生成测试：验证障碍物的生成间隔、不同难度的速度设置是否正确

- 状态切换测试：验证游戏三个状态的切换、游戏重置的逻辑是否正确

## 六、开发收获

通过本次大作业的开发，我收获了很多实践经验：

1. **模块化编程的实践**：将游戏拆分为输入、更新、绘制等独立模块，让代码的可读性和可维护性大幅提升，修改单个功能时不会影响其他模块，体会到了结构化编程的优势。

2. **游戏主循环的理解**：深入理解了游戏开发的核心主循环结构，以及帧率控制的重要性，明白了如何保证游戏在不同设备上的运行一致性。

3. **碰撞检测的原理与应用**：学习了 AABB 碰撞检测的基础原理，并且根据游戏的玩法需求做了定制化的优化，实现了不同的躲避方式，让游戏的玩法更加丰富。

4. **图形库的应用实践**：学会了使用 EasyX 图形库开发图形化程序，掌握了图片加载、UI 绘制、输入处理等基础图形开发技能，完成了大作业的图形化要求。
